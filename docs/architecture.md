## 1. 当前架构

**纯客户端直连云服务，没有自建后端。**

```
┌─────────────────────────────┐
│  Flutter App (PZ200 平板)   │
│  ┌─────────────┐            │
│  │  UI / 状态机 │            │
│  └──────┬──────┘            │
│         │                   │
│  本地存储：                 │
│  - sqflite（任务/语音备注） │
│  - SharedPreferences（偏好/缓存城市）│
│  - device_id (UUID)         │
│         │                   │
│  直连云服务（API Key 嵌入客户端） │
└─────────┬───────────────────┘
          │
   ┌──────┼──────────────┬──────────┐
   ▼      ▼              ▼          ▼
 豆包    讯飞 IAT      讯飞 TTS     高德
 (LLM)  (流式 ASR)   (流式 TTS)   (IP 定位)
 W1+    W2           W2          W2
```

W4a 已加入讯飞 AIKit 离线唤醒（"小治小治"）和设置页诊断，同样客户端直接接入；`AIKit.aar` 与 `aikit_resources/ivw` 已接入，首次运行需要联网激活 license，之后离线运行。资源目录优先使用官方 `android/app/src/main/assets/aikit_resources/`，桥接层也兼容旧目录 `android/app/src/main/assets/aikit/resource/`。AIKit 在进程内只初始化一次，普通停止只结束当前唤醒会话；唤醒触发后有 30 秒冷却期，避免 Flutter 生命周期恢复时立刻重启监听并抢占后续 ASR 麦克风。

当前真实进展：

- 对话、ASR、TTS、豆包联网问答主链路已接上
- 产品体验和稳定性仍在 W2 收尾，不能算完整完成
- 写入类闭环已进入回归验证阶段；离线唤醒已接入 SDK 和资源，待 PZ200 真机验收；长期记忆还没进入完成态

## 1.1 对话链路的正式分层（当前确定方向）

当前不能再把问题理解成“一个模型 + 几个 prompt”。正式方案是：

```
UI / ASR 输入
  ↓
任务理解层
  - 意图分类
  - 槽位补全
  - 上下文构建
  ↓
请求路由层
  - 公网信息 → 豆包 Responses API + web_search
  - 本地动作 → local tools / function calling
  ↓
回答渲染层
  - 文本
  - 结果卡
  - TTS
```

### 这一层不是推翻当前架构，而是补强

- **保留当前双链路**：公网继续走豆包内置联网，本地继续走本地 tools
- **新增任务理解层**：解决“什么时候补当前位置”“什么时候该走本地写入”“什么时候先给默认方案再继续细化”
- **不回退到单点公网接口堆砌**：不再把天气、汇率、新闻都做成一堆独立 API 适配器

### 第一版意图集合

| 意图 | 默认去向 |
|---|---|
| `general_qa` | 公网/模型回答 |
| `realtime_info` | 豆包联网 |
| `local_search` | 豆包联网 + 位置上下文 |
| `trip_planning` | 豆包联网 + 槽位补全 |
| `schedule_write` | 本地 tools + 二次确认 |
| `reminder_write` | 本地 tools + 二次确认 |

### 第一版槽位原则

- `trip_planning`：先关注 `origin / destination / date / duration`
- `local_search`：先关注 `location / category`
- `schedule_write`：先关注 `title / time`
- `reminder_write`：先关注 `content / time`

能自动补的就自动补，不能稳定补的就追问，但不要把所有问题都变成反问。

## 2. 这种架构的取舍

| 维度 | 直连（现状） | 自建后端 |
|---|---|---|
| 开发速度 | 快 | 慢 |
| 运维 | 零负担 | 要服务器 |
| **API Key 安全** | ⚠️ Key 嵌在客户端（APK 反编译可拿到） | Key 只在服务端 |
| 配额防刷 | ⚠️ 用户能盗刷 API 额度 | 服务端限流 |
| 多端同步数据 | 做不了 | 用户体系撑 |
| Prompt / 工具升级 | 要发新版 | 服务端热更 |
| 跨用户 RAG / 知识库 | 难做 | 服务端做 |

## 3. 业界惯例

| 阶段 | 通常做法 |
|---|---|
| 个人开发 / 自用 | 直连云（当前架构） |
| 内部工具 / 小团队 demo | 直连云 |
| **公开发布 / 商业化** | 必须有自建后端（哪怕只是薄代理层） |
| 多端 ToC 产品（豆包 App、华为小艺等） | 完整后端 + 用户体系 + 数据同步 |

## 4. 对 smart_workbench 的判断

| 场景 | 是否需要后端 |
|---|---|
| 只放自家平板 / PZ200 内置出货给企业客户 | 不必。设备不是公开分发，key 反编译风险可控 |
| 公开 App Store 上架 | **必须**加后端层 |
| 服务多个企业客户 | **必须**加后端层 |

## 5. 引入后端的三种渐进方案

如果哪天判断需要后端，按需求渐进：

### 方案 A — 最薄代理层（半天到 1 天）

仅做"App 请求 → 后端注入真 key → 转发给豆包/讯飞 → 流式回传 App"。

- 解决"API Key 暴露"风险
- 解决"用户盗刷"风险
- App 改动：把 base URL 从 `ark.cn-beijing.volces.com` 改成自家域名，去掉 .env 里的真实 key
- 后端栈推荐：Node + Express，或 Go + gin（流式转发用 SSE / WebSocket pipe）

### 方案 B — 中等（1-2 周）

在 A 基础上加：
- 设备/用户注册（device_id 挂账户）
- 配额限流（按设备/账户每日 token 上限）
- 简单的 prompt 远端配置（让 prompt 不用发新版改）

### 方案 C — 完整（数周）

- 用户体系（手机号/微信登录）
- 跨设备数据同步（任务、对话历史、偏好）
- RAG 知识库（用户上传文档、企业内部资料）
- 后台运营面板

## 6. 当前阶段建议

**MVP（W1-W4）继续直连云，不引入后端。**

理由：
- 节省精力，先验证产品形态
- W4 跑通完整体验后再做后端架构决策

**MVP 跑通后判断**：
- 只 PZ200 内置 → 不加
- 上 App Store / 多客户 → 立刻做方案 A，再按需求迭代到 B/C
