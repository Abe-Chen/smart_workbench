# 2. 业务背景与目标

## 2.1 业务背景

办公桌上的 45-60 岁高管，一天 8-12 个会议 + 客户拜访 + 电话，时间被切得很碎。常见痛点：
- 用手机日历漏看通知、看不清小字、打字慢
- 用本子记事翻不快
- 用通用语音助手不够专业，怕被同事看到「不像办公的人」
- 用秘书成本高，且自己已记不清「跟谁约了什么」

**核心矛盾**：他们需要一个永远在桌上的「类秘书」助理，不是聊天玩具，能听、能记、到点叫人。

## 2.2 产品目标

**北极星指标**：日均成功完成的助理任务数（创建/查询/修改/提醒）。

**MVP 验证目标**：
- 让目标用户**说一句话**就能完成日程的「记 / 查 / 改 / 删 / 完成 / 提醒」6 个核心动作
- 让目标用户**说一句话**就能完成备忘的「记 / 智能提醒 / 转日程」3 个核心动作
- 助理整体响应感觉「像个资深秘书」，不像机器人

## 2.3 成功标准

| 维度 | 标准 |
|---|---|
| 任务完成率 | 12 场景每个 ≥90% 一次说对 |
| 首字延迟 | 中位数 ≤ 3 秒 |
| 提醒触发准时 | 误差 ≤ 1 分钟 |
| 用户满意度（内测） | ≥ 7 / 10 |

# 3. 用户画像

## 3.1 核心用户

**张总，52 岁，某制造公司副总**

| 维度 | 特征 |
|---|---|
| 一天典型 | 6:30 起床，8:00 到办公室。上午 2-3 个内部会议，下午 2-3 个客户拜访，傍晚 1-2 个电话 |
| 工具习惯 | 飞书 / 微信 / 邮件，几乎不用专业生产力工具 |
| 设备 | 工位桌上 24 寸显示器 + 笔记本 + 手机 + 平板 |
| 痛点 | 老花眼，看不清手机小字；常常会议结束 5 分钟内忘记接下来的下一项 |
| 信任 | 信任秘书帮他记事；不信任「让 AI 自己做决定」 |
| 对桌面助理的预期 | 像秘书：听话、记得住、到点提醒、不主动开口 |

## 3.2 不是目标用户

- 技术爱好者 / 极客（他们想看到底层细节）
- 25-40 岁中层（他们用手机更熟练）
- Z 世代（他们要好玩）
- 一线工人 / 户外工作者（设备形态不匹配）

# 4. 用户故事与场景

按高频到低频排，每个场景用「作为...我希望...这样我可以...」格式。

## S1 创建日程（每天 5-10 次）
> 作为商务高管，我希望对桌面平板说「明天下午 3 点跟王总开会，提前半小时提醒我」，**助理就帮我记下日程并设好提醒**，这样我不用打开手机日历手动输入。

## S2 查询日程（每天 3-5 次）
> 作为商务高管，我希望问「明天有什么安排？」，**助理就读出明天的全部安排**，这样我下班前心里有数。

## S3 修改日程（每天 1-3 次）
> 作为商务高管，我希望说「把那个会议改到 4 点」，**助理基于上下文找到刚才的会议并改好**，这样我不用重复说会议名。

## S4 删除日程（每天 1-2 次）
> 作为商务高管，我希望说「那个会议取消」，**助理删除该日程并给我 5 秒钟撤销机会**，这样误删能补救。

## S5 创建备忘（每天 3-5 次）
> 作为商务高管，我希望说「记一下，张总让财务核 Q2 的款项」，**助理帮我存为备忘**，这样事情先存下来，回头再处理。

## S6 备忘智能提醒（每周 1-3 次）
> 作为商务高管，我希望对带时间表述的备忘（如「下周给客户发提案」）**助理自动设提醒，到点弹卡问我处理了吗**，这样我不会漏。

## S7 查天气（每天 1-2 次）
> 作为商务高管，我希望问「今天天气怎么样」，**助理弹卡显示当地天气**。

## S8 查汇率（偶尔）
> 作为商务高管，我希望问「美元今天多少」，**助理弹卡显示当前汇率**。

## S9 查附近吃饭（偶尔）
> 作为商务高管，我希望问「附近哪儿有湘菜」，**助理弹卡显示附近 1km 内候选**。

## S10 查新闻（偶尔）
> 作为商务高管，我希望问「最近有什么科技新闻」，**助理弹卡显示当日要闻**。

## S11 到点提醒（每天 5-10 次）
> 作为商务高管，我希望提醒时间到时**桌面平板弹起一张大卡 + 语音播报**，并提供「我知道了 / 再延 10 分钟 / 关闭」三个选项。

## S12 备忘转日程（每周 1-2 次）
> 作为商务高管，我希望对备忘说「转日程，下周三上午」，**助理把备忘升级为日程**。

# 5. 助理是怎么呈现给用户的

> 此节解决一个对评审重要的问题：助理在屏幕上到底有几个"位置"，每个位置承担什么？

## 5.1 5 个呈现位置

| 位置 | 在屏幕哪里 | 什么时候出现 | 承担的内容 |
|---|---|---|---|
| 小球 | 屏幕固定一角，永远在屏 | 始终在 | 助理的状态指示（待命 / 在听 / 在想 / 在答）|
| 底部回显条 | 屏幕中下方，弹起的横条 | 用户开口说话时 | 显示「我在听」+ 用户正在说的话（实时显示）+ 倒计时 + 取消按钮 |
| 顶部小卡 | 屏幕顶部，从上滑下 | 助理主动给建议时 | 主动建议（如创建日程后弹「要不要顺便加个提醒？」），点击可展开到抽屉看更多 |
| 全屏大卡 | 屏幕中央，占据大部分屏幕 | 助理答复 / 确认 / 错误 / 提醒触发时 | 助理"说话"的主要载体；7 种形态（详见 5.2） |
| 抽屉 | 用户上滑唤起，从底部弹起 | 用户主动上滑 | 看历史对话记录、看用户之前的消息流 |

## 5.2 全屏大卡的 7 种形态

| 形态 | 出现时机 | 是否自动消散 |
|---|---|---|
| 文字回答 | 短答复 / 边说边显示的占位 | 5 秒 |
| 信息卡 | 天气 / 汇率 / 时区时钟 / 附近吃饭 / 新闻 / 日程查询 | 5 秒 |
| 操作反馈 | 写入成功（创建 / 删除 / 完成）| 5 秒 |
| 追问卡 | 缺关键信息时主动问用户 | 不消散，等用户回答 |
| 确认卡 | 写入前请用户确认 | 不消散，等用户确认/取消 |
| 提醒卡 | 到点提醒 | 不消散，等用户操作 |
| 错误卡 | 出错兜底 | 5 秒 |

## 5.3 助理能识别的 7 类用户诉求

助理首先要听懂用户想做什么，目前能识别 7 类诉求（外加一个"不确定"兜底）：

| 诉求类型 | 用户表达样例 |
|---|---|
| 一般问答 | 「中秋节是几号」「人为什么会打哈欠」 |
| 实时信息查询 | 「今天天气怎么样」「美元今天多少」 |
| 附近搜索 | 「附近有什么湘菜馆」 |
| 出行规划 | 「明天去上海怎么走最快」 |
| 写日程 | 「明天下午 3 点开会」 |
| 设提醒 | 「下午 5 点提醒我接孩子」 |
| 记备忘 | 「记一下，张总让财务核 Q2 款项」 |

听不明白时，助理会用追问卡问一句，而不是瞎答。

# 7. 功能详细需求

每个功能含：用户故事 / 触发 / 交互流程 / 输入 / 输出 / 异常 / 文案 / 验收标准 / 依赖。

> **状态图例**：✅ 已实现 / 🟡 部分实现 / 🔴 未开始

## F1 一句话创建日程

| 项 | 值 |
|---|---|
| 优先级 | P0 |
| 状态 | ✅ |
| 关联场景 | S1 |

**用户故事**：作为商务高管，我希望说一句话就能记下日程，避免手动输入。

**触发方式**：用户唤醒（说「小治小治」）或长按麦克风，然后说带时间的指令（如「明天下午 3 点跟王总开会，提前半小时提醒」）。

**交互流程**：

```
用户对桌面平板说话
  ↓ 助理听清楚 + 自动忽略「啊嗯哈」语气词
助理判断这是「写日程」诉求
  ↓ 从用户的话里抽 4 个关键信息：标题 / 日期 / 起止时间 / 提醒
  ↓ 关键信息齐
弹确认卡（顶部「没问题我就这样记了哈」+ 4 行摘要）
  ↓ 用户说「确认」或点确认按钮
日程入库 + 设好到点提醒
  ↓
助理语音播报：「好的，已经放到日程里了。明天 15:00 跟王总开会。」
  ↓ 5 秒
大卡自动消散
```

**输入**：用户语音指令，至少包含标题或时间一项。

**输出**：本地多一条日程记录；若设了提醒，到点平板会弹起来；助理用语音和大卡同时反馈。

**标准文案**：
- 进度球转圈时：「我整理一下信息」
- 确认卡标题：「没问题我就这样记了哈」
- 创建成功：「好的，已经放到日程里了。[日期] [时间] 「[标题]」。」

**异常处理**：

| 异常 | 助理表现 | 文案 |
|---|---|---|
| 全部信息都缺 | 弹追问卡 | 「可以。这个日程是什么，安排在什么时候？」 |
| 缺标题 | 弹追问卡 | 「这条日程叫什么？」 |
| 缺时间 | 弹追问卡（给出兜底）| 「[日期]几点开始？也可以说"全天"。」 |
| 用户取消 | 关确认卡 | 「好，这次先不记。」 |
| 写入失败 | 弹错误卡 | 「这次没记上：[原因]。要不再来一次？」 |

**验收标准**：

- AC1 用户说一句完整指令（标题+日期+时间+提醒），助理 3 秒内弹出确认卡
- AC2 确认卡显示标题、日期、时间、提醒 4 行
- AC3 用户说「确认」后日程入库且设好到点提醒
- AC4 用户说「取消」时不入库
- AC5 标题或时间缺失时弹追问卡
- AC6 文案完全符合标准文案

**依赖**：F12 / F13 / F14 / F6。

## F2 多维度查询日程

| 项 | 值 |
|---|---|
| 优先级 | P0 |
| 状态 | ✅ |
| 关联场景 | S2 |

**用户故事**：作为商务高管，我希望随时问「今天 / 明天 / 这周 / 接下来 / 现在该做什么」，助理给我准确的列表。

**支持的查询维度**：

| 维度 | 用户表达样例 |
|---|---|
| 当日全部 | 今天有什么 |
| 时段筛选 | 今天下午有空吗 / 上午有什么 |
| 跨天范围 | 这周 / 最近 7 天 |
| 状态筛选 | 已完成的 / 已删的 |
| 正在进行 | 我现在该做什么 |
| 下一项 | 下一项是什么 |
| 跨日上限 | 最近 N 天，含全量统计 + 按日分组 + 截断显示 |
| 空查询 + 下一项兜底 | 今天下午没安排吧 |

**交互流程**：

```
用户问 → 助理判断为查日程 → 查本地数据
  ↓
有结果：弹列表大卡 + 语音摘要播报「[范围]有 N 个安排：…」
无结果：弹大卡「[范围]没有安排」+ 若有下一项追加「下一项是…」
  ↓ 5 秒
大卡自动消散（触屏点卡可延长）
```

**标准文案**：见 MVP 终态定义文档 §2.3 S2 节。

**异常处理**：

| 异常 | 助理表现 |
|---|---|
| 查询条件歧义（如「这周」是否含今天） | 默认含今天 + 第一句话说明 |
| 多条截断 | 「[范围]共有 N 个安排…屏幕上只放摘要。」 |

**验收标准**：

- AC1 8 种查询维度都能返回正确结果
- AC2 列表按时间正序，全天事项排最上面
- AC3 4 条以上自动摘要（"今天接下来还有 N 个"）
- AC4 空结果时若有下一项必须播报
- AC5 文案符合规范

**依赖**：F12 / F14。

## F3 上下文修改日程

| 项 | 值 |
|---|---|
| 优先级 | P0 |
| 状态 | ✅ |
| 关联场景 | S3 |

**用户故事**：作为商务高管，我希望基于上下文说「改到 4 点」就能改前一条日程，不用重复说标题。

**支持修改字段**：标题 / 日期 / 时间 / 提醒 / 重复。

**交互流程**：

```
用户说「把那个会议改到 4 点」
  ↓
助理基于上下文找最近的会议
  ↓ 从用户话里抽出要改的内容（时间=16:00）
弹确认卡「这样改吗」+ 前后对比（15:00 → 16:00）
  ↓ 用户确认
更新数据 + 取消旧提醒 + 设好新提醒
  ↓
助理语音播报「好的，改好了。」
```

**异常处理**：

| 异常 | 助理表现 |
|---|---|
| 没说要改哪个 | 「您想改哪一条？」 |
| 多条匹配 | 列候选「您是说哪个？」 |
| 改完字段都没变 | 「这条没有变化哦。」 |
| 用户取消 | 「好，这次先不改。」 |

**验收标准**：

- AC1 「改到 X 点」基于最近上下文能正确找到目标
- AC2 改时间后新提醒按新时间触发，旧提醒取消
- AC3 多条匹配时列候选不直接执行
- AC4 文案符合规范

**依赖**：F12 / F14。

## F4 删除日程 + 5 秒撤销

| 项 | 值 |
|---|---|
| 优先级 | P0 |
| 状态 | 🟡 删除已实现，撤销条 UI 未接 |
| 关联场景 | S4 |
| 待开发工作量 | 0.3 天 |

**用户故事**：作为商务高管，我希望删错日程时能立刻撤销，避免重新创建。

**待开发部分**：在删除成功的反馈大卡底部加 5 秒撤销条（类似 Material 的浮动提示形态），点击后恢复数据 + 重新设好原来的提醒。

**交互流程**：

```
用户说「那个会议取消」→ 助理弹「这条要删吗」→ 用户确认
  ↓
软删除 + 取消提醒 + 反馈大卡 + 底部撤销条
  ↓ 5 秒内用户点撤销
恢复数据 + 重新设提醒 → 助理语音「好的，撤销了。」
  ↓ 5 秒未操作
撤销条消失，删除生效
```

**异常处理**：

| 异常 | 助理表现 |
|---|---|
| 没说删哪个 | 「想删哪一条？我先查一下。」 |
| 多条匹配 | 列候选 |
| 撤销窗口外才点 | 撤销按钮已消失 |

**标准文案**：
- 确认卡标题：「这条要删吗」

| 触发后 | 恢复数据 + 重新设好原提醒 + 助理语音「好的，撤销了。」 |
| 窗口外 | 撤销条消失，操作不可逆 |

## 8.4 错误兜底规范

| 错误类型 | 触发条件 | 兜底动作 |
|---|---|---|
| 首字超时 | 问出去 8 秒还没开始回话 | 弹错误卡 + 重试按钮 |
| 半天没说话 | 已开始回话但 10 秒没新内容 | F15 三按钮 |
| 硬超时 | 总耗时 ≥ 45 秒 | 弹错误卡 + 重试按钮 |
| 写入失败 | 本地存储写不进去 | 弹错误卡 + 重试按钮 |
| 语音播报失败 | 主语音不可用 | 静默切到备用语音 |
| 语音识别失败 | 没听清 | 静默重试一次，再不行兜底「您再说一次吧」 |

## 8.5 等待期可控规范

适用：F14 边说边显示 + F15 兜底。

| 等待阶段 | 用户感知 |
|---|---|
| 0-3 秒 | 球转圈 |
| 3-10 秒 | 球转圈 + 进度文案逐步刷新（如「我查查哈」「查到了」）|
| 10 秒起 | 大卡底部出三按钮（F15）|
| 45 秒 | 自动兜底错误卡（F16）|

# 10. 非功能性需求

## 10.1 性能

| 指标 | 目标 |
|---|---|
| 首字延迟（中位数） | ≤ 3 秒 |
| 提醒触发误差 | ≤ 1 分钟 |
| 助理被用户打断响应 | ≤ 200 毫秒 |
| 创建/查询操作端到端 | ≤ 5 秒（从用户说完到大卡显示）|
| 本地数据查询 | ≤ 100 毫秒（30 天数据范围）|

## 10.2 可用性

| 场景 | 表现 |
|---|---|
| 网络断 | 助理告知「网有点慢」+ 重试按钮；本地动作（增删改查）继续可用 |
| 主语音失败 | 静默切到备用语音 |
| 语音识别失败 | 静默重试一次后兜底「您再说一次吧」 |
| 写入失败 | 弹错误卡 + 重试 |

## 10.3 可访问性

| 维度 | 要求 |
|---|---|
| 远距离阅读 | 1m 距离能看清关键字段（数字、标题、时间）|
| 色弱兼容 | 所有状态不仅靠颜色区分，配图标 + 文字 |
| 语音替代 | 所有视觉信息有语音等价播报 |

## 10.4 隐私

- 用户日程 / 备忘 / 偏好全部本地存储
- 仅必要的查询请求上传到云端（如「今天天气」会带城市名）
- 录音不持久化（说完就丢）
- 数据不上传服务端、不做账号绑定（MVP 阶段）
