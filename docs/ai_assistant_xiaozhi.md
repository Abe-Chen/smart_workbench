## 0. 关键决策摘要（已对齐）

| 维度 | 决策 |
|---|---|
| 用户场景 | 个人桌面 ToC，先不做账号 |
| 设备身份 | 本地生成 UUID 作为 device_id，作为豆包 API 的 `user_id` |
| 在线/离线 | LLM/ASR/TTS 在线（豆包/火山）；唤醒词离线 |
| LLM 服务 | 豆包大模型（火山引擎方舟） |
| ASR | 讯飞 IAT（WebSocket 流式，纯 Dart 实现） |
| TTS | 讯飞在线语音合成（WebSocket 流式，纯 Dart 实现） |
| 联网搜索（天气/新闻/酒店等） | 走豆包内置联网能力 |
| 任务理解 | **应用层做“意图分类 + 槽位补全 + 上下文构建”**，不只靠 prompt |
| 唤醒词方案 | **讯飞 AIKit 语音唤醒（Android 离线 SDK）**，自定义唤醒词"小治小治" |
| 讯飞凭据复用 | 唤醒 / ASR / TTS 共用一套 XF_APP_ID + XF_API_KEY + XF_API_SECRET |
| 写入数据前确认 | 必须二次确认 |
| 多轮记忆 | 跨会话 + 长期偏好 |
| 唤醒方式 | 语音"小治小治" + 单击悬浮球 + 长按悬浮球 |
| 输出方式 | 页面显示 + 语音播报双通道 |
| TTS 默认行为 | **语音入口默认播报 / 文字入口默认静默**；提供全局四模式（auto / always / shortOnly / silent）+ 抽屉本会话静音 toggle。详见 §6 |
| 架构边界 | **公网信息继续走豆包内置联网；本地动作继续走本地 tools，不回退到一堆单点天气/汇率接口** |

## 1. 产品角色

- **名称**：小治
- **可视化身**：当前底部导航栏中央的悬浮球（蓝紫渐变 + 笑脸光晕），延续作为小治形象
- **人设**：贴心、简短、能动手；不啰嗦不卖萌；像桌面上的高效同事
- **说话风格**：口语短句，2–3 句讲完。需要明细时才展开成卡片。

## 1.1 当前实际进展（2026-05-08，按代码核对修订）

### 已落地

- 抽屉对话主链路已可用，支持文字问答（`assistant_controller` + `assistant_drawer`）
- 讯飞 IAT / TTS 已接入，支持点击听音和长按说话；TTS 可中断、可重播、可切音色
- 豆包公开信息问答已切到 `Responses API + web_search`，并支持 `previous_response_id` 延续上一轮话题
- 抽屉 / 卡片两种回答形态已基本成型（`AssistantReplySurface.compactCard / drawer`）
- TTS 音色切换、试听、错误提示、断网超时提示已补到一版
- **8 秒开放麦 + 5 秒持续对话 + 倒计时圈已实现**（控制器 `_kOpenMicWait / _kFollowUpWindow` + ticker；UI 层 `_CountdownBadge` + 球外环 `countdownProgress`）。验收阶段只需做真机 QA
- **请求路由 v0 已实现**：`assistant_request_router` 做规则级 public/local 分流、公网话题延续判断（"那北京呢/那明天呢"系列）；`_buildPublicQuery` 在天气/附近类问题里自动补当前城市
- **本地任务链路话术层已补强**：新增 `AssistantCopywriter` 集中管理缺字段、查询结果、确认、完成、成功、失败、取消等微文案；controller 不再直接散落机器人式提示
- **助手 UI 视觉层已优化**：`AssistantOverlay` 增加屏幕边缘流动光效；`AssistantBall` 按 idle/listen/think/answer/confirm/error 切换渐变、光晕和内部动效；听音胶囊、轻回答卡、确认卡、聊天气泡已改成更贴近系统级助理的视觉层级。业务状态机、ASR/TTS、工具调用和数据库写入未改。
- **任务型对话可靠性 P0 已补强**：新增确定性本地任务指令层，优先处理“明天有什么安排”“明天下午 5 点开会”“把明天 3 点会议改成 4 点”“删除明天下午 3 点需求讨论会”“需要提醒”等高频任务；写入类请求必须先进入确认卡，确认前不写数据库；模型如果没有真实 tool call，不能只用文字声称“已创建/已删除”。

### 未完成

## 1.3 任务型对话可靠性方案（2026-05-08 补充）

### 背景问题

真机/手动测试暴露出的问题不是 UI，而是任务型对话管理不够严格：

- 槽位补全期间，用户说“查一下明天的天气”会被错误当成日程标题
- 查询、修改、删除过度依赖模型自由选择工具，容易查不到真实本地数据
- 写入动作存在“文字说成功，但数据库没变化”的风险
- 多轮确认态里，“需要提醒”这类补充信息没有被继续接住

### 设计原则

行业内成熟做法基本一致：

- **意图和槽位要分开处理**：先判断用户是在创建、查询、修改、删除，还是换了一个新问题
- **槽位补全要允许打断**：用户没有回答标题而是问天气时，应先处理天气，不能把天气问题塞进标题
- **写入前必须确认**：创建、修改、删除都先展示确认卡，确认后才执行工具
- **执行结果以工具返回为准**：没有真实 tool call 或工具返回失败时，不能说“已创建/已删除”
- **查找目标先查候选再匹配**：修改/删除必须先查本地日程，再按日期、时间、标题/泛称做匹配；多候选要让用户选择

### 已落地的 P0/P1 范围

- 创建打断：`创建一个明天下午 5 点的日程` 后，如果用户说 `查一下明天的天气`，会清掉草稿并走天气查询，不会生成“查一下 的天气”日程
- 自然创建：`明天下午 5 点开会`、`帮我加一个明天下午3点客户拜访` 会进入创建确认卡，不会直接写入
- 泛化自然创建：只要用户说出“明确日期 + 明确时间 + 具体事项”，例如 `明天早晨 9 点出差去石家庄`、`明天上午 9 点和客户交流产品对接的问题在客户现场`，都优先进入日程确认卡；不再只依赖“开会/拜访”少数关键词
- 查询排除：如果用户明确在问 `天气/新闻/汇率/价格`，或说 `怎么样/查一下/告诉我` 等查询句式，即使句子里有日期时间，也继续走公网查询，不误创建日程
- 本地查询：`明天有什么日程安排` 直接调用 `query_tasks`，由 App 组织回复，不让模型自由回答“查不到”
- 修改匹配：`把明天3点的会议改成4点` 会先查明天日程，用 3 点附近 + “会议/会/讨论会”等泛称匹配目标，再进入修改确认卡
- 多候选续接：如果 3 点附近有多条会议，小治会列出候选项；用户继续说 `第一条` / `第二条` 时，会沿用上一轮上下文进入修改/删除确认，不重新当成普通问题理解
- 删除匹配：`删除明天下午3点的需求讨论会` 会先查候选并进入红色删除确认卡，确认后才软删除
- 提醒补充：创建确认态下说 `需要提醒`，会给当前待确认日程补上默认“提前 10 分钟提醒”，流程不丢失
- 成功话术防护：本地写入意图如果没有真实工具调用，App 会拦截“已记录/已删除”等文本式成功话术
- 查询话术：`今天/明天有什么安排` 改成自然摘要，例如“明天有 2 个安排：- 15:00 需求讨论会”，不再输出“查到 2 条、1”这类机械表述

### 后续仍需补强

- 多候选选择 UI：当同一时间有多条会议时，需要可点击的候选列表，而不是只用文本让用户再说一遍
- 更完整的时间语义：`下午晚点`、`三点左右`、`下周三`、`本周五`、农历/节假日暂未完整覆盖
- 重复日程和提醒：`每周一提醒`、`每天 9 点`、`提前一天提醒` 需要独立的重复/提醒槽位状态机
- 真实端到端验收：需要在 PZ200 真机上验证确认卡、数据库刷新、看板同步和语音多轮输入

## 1.2 整体方案补强（防止后续开发跑偏）

### 正式结论

- 不能只靠系统提示词让小治变成“任务型助理”
- 也不能把公网能力继续拆回“天气接口 / 汇率接口 / 新闻接口”这种单点堆砌
- 正确方向是：**保留当前双链路架构，在它前面补一层“任务理解与槽位补全”**

### 保持不变的部分

- **公网信息链路**：继续走豆包 `Responses API + web_search`
- **本地能力链路**：继续走本地 function calling / tools（任务、日程、提醒、位置、TTS）
- **写入前确认**：继续保留，不能因为任务理解更强就跳过确认
- **短答卡片 / 长答抽屉 / TTS 分场景播报**：继续沿用现有产品方案

### 新增的一层：任务理解与槽位补全

在用户输入和当前双链路路由之间，增加一层轻量的应用层理解：

```
用户输入
  ↓
任务理解层
  - 意图分类
  - 槽位补全
  - 上下文构建
  ↓
请求路由层
  - 公网问答 → Responses API + web_search
  - 本地动作 → local tools / function calling
  ↓
回答生成与渲染
  - 文本
  - 结果卡
  - TTS
```

### 这层要解决什么问题

- 用户问天气时，会补当前位置
- 用户问“去成都出差怎么安排”时，也应该能补“出发地 / 日期 / 天数”这类关键信息
- 用户问“帮我加个明天下午提醒”时，不应走公网搜索，而应直接落到本地写入链路
- 用户连续追问“那北京呢”“那明天呢”“那从上海出发呢”时，不应全靠模型自由发挥

### 意图分类的最小集合

第一版先只做最小集合，不一开始铺太大：

| 意图 | 说明 | 默认走向 |
|---|---|---|
| `general_qa` | 普通常识问答、解释、建议 | 公网/模型回答 |
| `realtime_info` | 天气、汇率、新闻、价格、营业时间等实时信息 | 豆包联网 |
| `local_search` | 附近、周边、本地餐厅/酒店/路线/通勤 | 豆包联网 + 位置上下文 |
| `trip_planning` | 出差、旅行、路线、行程安排 | 豆包联网 + 位置/槽位补全 |
| `schedule_write` | 创建/修改/删除日程、会议、待办 | 本地 tools + 二次确认 |
| `reminder_write` | 创建/修改/删除提醒 | 本地 tools + 二次确认 |

### 第一版槽位规则

先只做最低必要槽位，不追求一次做全：

| 意图 | 必要槽位 | 可默认补的槽位 | 缺失时策略 |
|---|---|---|---|
| `trip_planning` | `origin` / `destination` | `origin` 可用当前位置补；`duration` 可先假设 2 天 1 晚 | 缺少日期/天数时先给默认方案，再提示可细化 |
| `local_search` | `location` / `category` | `location` 可用当前位置补 | 位置拿不到时反问“想查哪里的？” |
| `schedule_write` | `title` / `time` | 无 | 时间不清楚时必须追问或进入确认前编辑 |
| `reminder_write` | `content` / `time` | 无 | 时间不清楚时必须追问 |

### 位置上下文原则

- 位置不是只给天气用的
- `local_search`、`trip_planning`、路线、通勤、酒店、美食这类问题，都可以复用当前位置
- 只有在用户没有明确给出城市/出发地时，才主动补当前位置
- 一旦用户明确说了“北京 / 成都 / 从上海出发”，就以用户显式信息为准

### Prompt 的职责边界

- **Prompt 负责行为原则**：该短答还是展开、何时说明不确定、何时引用实时数据
- **应用层负责稳定性**：这是什么任务、缺什么槽位、是否该补当前位置、是否该走本地写入
- **不要把稳定性问题都丢给 prompt**

### 明确不做的事

- 不把公网实时信息重新拆成一堆硬编码单点接口
- 不让模型在没有本地执行能力时假装“已经创建成功”
- 不在 W2 阶段把任务理解层扩成一个过大的 NLU 系统

## 2. 唤醒方式

| 方式 | 行为 | 备注 |
|---|---|---|
| 语音 "小治小治" | 后台 KWS 持续监听，触发后自动进入听音态 | 低功耗常驻；技术选型见 §8 |
| 单击悬浮球 | 进入听音态等待 8 秒 + 默认打开右侧抽屉文字输入模式 | 无需说话也能用 |
| 长按悬浮球（>0.3s） | 听音态 + 实时音波反馈，按住说话松手结束 | 类似微信语音 |

**持续对话**：助手回答完后保持半唤醒 5 秒，期间用户直接说话不用重新唤醒。屏幕角标显示倒计时圈圈，5 秒无声音自动收起。

## 3. 状态机

```
                    ┌──────[唤醒]─────┐
                    │                 ↓
   静态(idle) ──> 听音(listen) ──[识别完]──> 思考(think) ──> 回答(answer)
       ↑               │                                       │
       │               └──[取消]──> 静态                        │
       │                                                       │
       │                              ┌─[需要写数据]── 操作确认(confirm)──┐
       │                              │                                  ↓
       └────────[5秒无对话/收起]──────┴────[执行完]────────────────────────┘
```

### 状态 1 · 静态（idle）

- **球**：当前蓝紫渐变球 + 2 秒呼吸光晕（alpha 0.7→1.0），表示"在岗"
- **屏幕**：无 AI 痕迹

### 状态 2 · 听音（listen）

- **球**：放大 1.15 倍；外层 3 圈彩色涟漪（蓝→青→紫渐变）随麦克风音量呼吸
- **屏幕边缘**：4–6px 渐变光带从悬浮球处发出，沿屏幕周长流动 1 圈后稳定为静态光环（华为小艺/Siri 风格）
- **屏幕中下方胶囊条**：`我在听... | 实时 ASR 转写文字 | ✕`
  - 转写文字按词逐字浮现，背景半透明黑
- **取消方式**：点 ✕ / 点屏幕外区域 / 5 秒无声音

### 状态 3 · 思考（think）

- **球**：内部三粒子绕中心旋转，外层涟漪消失
- **胶囊条**：`小治正在想...` + 3 点跳动加载动画
- **屏幕边框光带**：保留，但流动速度减半，饱和度下降

### 状态 4 · 回答（answer）

按内容长度自动分两种呈现：

#### 4a. 简短回答 → 底部卡片（默认带 TTS 播报）

适用：≤2 句话总结、可卡片化的查询结果（天气/新闻/搜索/查询日程）。

- 球：内部显示 TTS 音频波形（同 TTS 实时音量）
- 底部胶囊**展开为卡片**（200–300 高），从底部上滑动画入场
  - 头部：小治头像 + 文字回答（打字机逐字显现）
  - 主体：场景化卡片（天气卡 / 新闻卡 / 日程卡 / 搜索结果卡）
  - 尾部：`还需要什么？` + 5 秒倒计时圈
- TTS 只念头部一句话，卡片明细不念

#### 4b. 长内容/连续对话 → 右侧抽屉（默认静默，可手动开 TTS）

适用：闲聊、长答案、需要查看历史的对话。

- 屏幕右侧滑出 40% 宽抽屉，**看板可见可交互**（不被遮、不变暗）
- 抽屉头部：小治头像 + 标题
- 抽屉主体：对话流（用户气泡 + 小治气泡 + 内嵌卡片）
- 底部：输入框 + 语音按钮 + 关闭按钮 + 静音/播报切换图标
- 球嵌在抽屉顶部
- 关键差异化：**AI 改了日程，左侧看板对应卡片实时变** — 这是相对豆包 App 最强的体验

## 6. 语音播报（TTS）规则

### 6.1 当前已落地（W2 阶段 1 优化后）

- **念什么**：只念回答头部一句话总结，不念卡片明细。Phase 2 会改成按句流式整段播报。
- **可中断**：在 TTS 播报中开新一轮听音（点球 / 唤醒）→ 立即停止；W4 唤醒桥接好后支持 barge-in。
- **音色**：当前讯飞五个音色二选一（聆小璇 / 聆玉昭 / 聆小玥 / 聆玉言 / 聆飞逸），设置页可切换。
- **语速**：四档 0.8 / 1.0 / 1.2 / 1.5，对应讯飞 `business.speed` 30 / 50 / 65 / 85。

### 6.2 播报策略：三层叠加（高优先级到低）

```
第一层（最高）：本会话静音 toggle  ── 抽屉头按一下，仅本对话生效
                                      不写入 settings，clearConversation 重置
       ↓ 未触发
第二层：全局播报模式（settings 持久化）
   - silent     → 永远不播
   - always     → 全播
   - shortOnly  → 仅短答（compactCard）
   - auto（默认）↓ 进第三层
       ↓
第三层（auto 模式下的场景默认）：
   - 短答卡片（compactCard）       → 播
   - 长答抽屉（drawer）+ 语音入口 → 播     ← 改进：drawerVoice / quickVoice
   - 长答抽屉（drawer）+ 文字入口 → 不播   ← 保持：drawerText
   - 无承载面（none）               → 不播
```

### 6.3 这个策略解决的旧痛点

| 旧行为 | 新行为 |
|---|---|
| 抽屉里语音问问题 → 不播报（反直觉） | 播报（auto 模式默认） |
| 快速语音问长问题 → 被甩到抽屉里不播 | 播报（auto 模式默认） |
| 设置里没"一键全局静音" | 模式 = silent / 抽屉 toggle |
| TTS 错误显示在听音错误位 | 独立 `ttsError` 通道 + 抽屉单独提示 |

### 6.4 待 Phase 2/3 解决（未做）

- ❌ TTFA（首字延迟）从"整段合成完才播"降到 < 500ms（句级流式 / 边收边播）
- ❌ 长答只播第一句 / 120 字截断 → 全段流式
- ❌ 中间暂停 / 跳过按钮
- ❌ TTS 缓存（重播不必重合成）
- ❌ 切音色立即播 sample（当前需手动点试听）

## 7. 长期记忆与设备身份

### 数据模型（sqflite 新增）

```sql
-- 设备 ID（首次启动生成 UUID v4，存 SharedPreferences 即可，单条无需建表）

-- 对话历史
CREATE TABLE ai_conversations (
  id INTEGER PRIMARY KEY,
  started_at TEXT,
  ended_at TEXT,
  summary TEXT  -- 由 AI 主动生成的会话标题/摘要
);

CREATE TABLE ai_messages (
  id INTEGER PRIMARY KEY,
  conversation_id INTEGER,
  role TEXT,      -- 'user' | 'assistant' | 'system' | 'tool'
  content TEXT,   -- 文字内容
  function_call TEXT, -- JSON：function calling 调用记录
  ts TEXT,
  FOREIGN KEY(conversation_id) REFERENCES ai_conversations(id)
);

-- 长期偏好库（小治记住的用户信息）
CREATE TABLE ai_user_preferences (
  key TEXT PRIMARY KEY,         -- "wife_name" / "home_city" / "morning_routine"
  value TEXT,                   -- "小芳" / "上海" / "07:00-08:30"
  source TEXT,                  -- 'explicit' (用户主动告诉) | 'inferred' (AI 推断)
  confidence REAL,              -- 0.0–1.0
  updated_at TEXT
);

-- 写入动作审计（用户可看历史/可撤销）
CREATE TABLE ai_action_log (
  id INTEGER PRIMARY KEY,
  conversation_id INTEGER,
  action TEXT,    -- 'create_task' / 'update_task' / 'delete_task' / 'remember'
  payload TEXT,   -- JSON：完整请求
  result TEXT,    -- JSON：执行结果
  ts TEXT
);
```

### 工作机制

- **每轮对话开始**：把 `ai_user_preferences` 中 confidence > 0.7 的偏好作为 system message 注入豆包 API
- **每轮对话结束**：AI 用 function call `remember(key, value, source)` 主动写偏好
- **设备 ID 用法**：作为豆包 API 的 `user_id` 参数，让豆包侧维持长上下文（如果其 API 支持）
- **隐私**：用户可在"我的" tab 下查看 / 清除对话历史和偏好库

## 8. 技术栈对应豆包能力

| 模块 | 服务 | 备注 |
|---|---|---|
| LLM 对话 | 豆包大模型（doubao-pro-32k 或 1.5-pro） | OpenAI 兼容 API，支持 function calling |
| 联网搜索 | 豆包内置联网（API 参数格式需查证最新方舟文档） | 天气/新闻/酒店都靠它 |
| ASR | 讯飞 IAT（WebSocket 流式，wss://iat-api.xfyun.cn/v2/iat） | 纯 Dart `web_socket_channel` 实现，鉴权 HMAC-SHA256 |
| TTS | 讯飞在线语音合成（WebSocket 流式，wss://tts-api.xfyun.cn/v2/tts） | 同上鉴权方式，跟 ASR 共用 HMAC 签名工具 |
| 唤醒词 "小治小治" | **讯飞 AIKit 语音唤醒**（Android 原生 SDK） | 见 §8.1 |
| 设备身份 | 本地 UUID + device_info_plus（已在依赖里） | 不做账号 |

> **音色备选回退**：如果未来希望用豆包专属"灿灿"音色，TTS 切到火山引擎流式 TTS（WebSocket + 4 字节 header 自定义二进制帧）。届时通过更换 `TextToSpeechClient` provider 实现切换，业务代码不动。新增约 200 行编解码代码。

### 8.1 讯飞唤醒词集成要点

> 文档来源：https://www.xfyun.cn/doc/asr/AIkit_awaken/Android-SDK.html

**核心特性**：
- 纯离线运行（首次联网激活获取 license，之后完全离线工作）
- 支持自定义唤醒词，通过配置文件 `小治小治;nCM:300;` 形式（UTF-8）
- 提供 Android / iOS / Linux / Windows 原生 SDK
- 鉴权用 AppID + ApiKey + ApiSecret（讯飞开放平台获取）

**集成路径**（坑点提示）：
- ⚠️ **没有官方 Flutter 包**，必须自己写 Flutter ↔ Android 原生的 MethodChannel 桥接层
  - Android 侧：导入 `AIKit.aar`，初始化 + 加载唤醒词配置文件 + 监听回调
  - Flutter 侧：通过 `EventChannel` 接收唤醒事件，触发 §3 状态机的"唤醒 → 听音态"
  - 后续 iOS 也得分别写一遍（如果产品支持 iOS）
- 资源：`resource/` 文件夹放 assets，初始化时复制到 workDir
- 自定义关键词调参：`nCM`（词门限）、`nSubCM`（字门限）控制识别灵敏度，需要真机调

**工作流**：
```
1. 注册讯飞开放平台 → 创建应用 → 获取 AppID/ApiKey/ApiSecret
2. 在控制台为该 AppID 开通"语音唤醒"能力 + 配置自定义唤醒词"小治小治"
3. 下载 AIKit Android SDK（.aar + resource/）
4. 集成到 android/app/，写 MethodChannel 桥接
5. 首次启动时联网 license 激活，缓存到 workDir
6. 之后离线运行，唤醒后通过 EventChannel 通知 Flutter 层
```

| 周 | 目标 | 验证标准 |

> ⚠️ W4 因为讯飞无官方 Flutter 包需要写原生桥接，工作量比纯 Dart 改动重，可能需要拆分成 W4a（讯飞 Android 桥接独立调通）+ W4b（接到状态机）。

### W2 验收清单（未全部通过前，不算完成）

| 项 | 验收标准 |
|---|---|
| 听音链路 | 单击听音、长按说话、取消、结束，都行为一致，不误发、不漏发 |
| ASR 稳定性 | 实时转写正常，弱网/失败时提示清楚，不出现卡死感 |
| TTS 稳定性 | 能播、能停、能重播、能切音色、试听结果可区分 |
| 公网问答 | 有网能答，断网或超时提示短而清楚，不抛大段英文异常 |
| 展现逻辑 | 短答卡片、长答抽屉、等待态、错误态都不误导用户 |
| UI 体验 | 边缘光效、听音胶囊、抽屉输入框、键盘避让、看板共存不互相遮挡 |
| 基本回归 | 设置页、看板、日程页、助理交互互不打架 |

### 当前建议的开发顺序

1. 先按 W2 验收清单逐项收尾，直到能明确判定"W2 完成"（持续对话/倒计时圈代码已就绪，按清单做真机 QA 即可）
2. 进入 W3a：在已有 `assistant_request_router` 基础上补意图显式标签（六意图）+ 槽位树（`title / time / origin / destination / date / duration / content`）
3. 再做 W3b：在 `tool_registry` 注册任务/日程/提醒写入工具 + 写入前确认卡 + sqflite 真实写入 + 看板联动刷新
4. 然后做 W4a：讯飞 AIKit Android 唤醒桥接（`MainActivity` 改成 `FlutterActivity` 子类 + MethodChannel/EventChannel）
5. 最后补 W4b：长期记忆三张表（`ai_conversations / ai_messages / ai_user_preferences / ai_action_log`）+ 偏好注入到 system prompt
